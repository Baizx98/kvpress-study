# BlockWisePress 跨层分数残差方案

## 1. 问题背景

前一轮 `RULER 0.7` 的失败案例分析和轻量 `token correction` 定向验证，给出了两个比较明确的信号：

1. 当前 `BlockWisePress` 的主要问题，不只是块内少数关键 token 被稀释。
2. 更大的问题可能是高压缩下块排序在层间不够稳定，导致远距离关键块和支持块的召回抖动较大。

与此同时，刚刚验证过的 `lightweight token correction` 并没有带来提升，反而略有下降。这说明：

- 仅仅在当前摘要框架上补一层“块内高范数 token 校正”，并不能有效解决主要矛盾。

因此，下一步更值得尝试的是一个更简单、也更符合系统约束的方向：

- 跨层分数残差 / 平滑

## 2. 核心假设

### 假设

对于同一个样本、同一个生成步，在相邻注意力层之间：

- 真正重要的块集合不会发生剧烈突变
- 关键块的重要性排序应该具有一定层间连续性

如果只使用当前层的即时块分数：

- 高压缩下容易因为单层噪声而把远距离关键块挤出保留集合

如果把上一层的块分数作为一个很轻的残差项引入当前层：

- 有望稳定块排序
- 减少远距离热点块在层间被反复抖动删除的概率

## 3. 方法设计

### 3.1 总体思路

在 `BlockWisePress` 中，保留当前“块摘要 + query-aware 打分”的主路径不变，只在最终块分数处加入上一层块分数：

```text
score_l = (1 - w) * current_score_l + w * score_{l-1}
```

其中：

- `current_score_l` 是当前层按现有摘要机制算出的块分数
- `score_{l-1}` 是上一层的最终块分数
- `w` 是跨层残差权重

### 3.2 为什么采用线性残差

这个形式有几个优点：

1. 实现简单  
   不引入新的复杂状态机，不需要新 attention，不需要额外 token 级计算。

2. 开销低  
   只需要缓存上一层一个块分数向量，元数据量很小。

3. 适合未来卸载系统  
   这本质上只是热度分数的层间平滑，不会破坏块级摘要常驻 GPU 的设计。

### 3.3 作用范围

这次先只做：

- 同一次 forward 内，相邻层之间的残差连接

暂时不做：

- 跨 decode step 的时间平滑
- 更复杂的 EMA 历史项
- 按层学习或自适应权重

### 3.4 与上一轮 token correction 的关系

为了让实验结论更干净，这一版优先做“残差机制单独验证”：

- 保留 `token correction` 代码接口
- 但默认关闭 `token_correction_weight`
- 先只看跨层残差能否改善 `RULER 0.7`

这样后续如果残差有效，再考虑是否和 token correction 组合。

## 4. 默认参数建议

### 4.1 当前建议

- `cross_layer_score_residual_weight = 0.2`

理由：

- 太小，可能稳定作用不明显
- 太大，会把上一层错误排序直接传递下来
- `0.2` 是一个比较稳妥的初始点，表达“当前层为主，上一层做轻量平滑”

### 4.2 边界策略

仅当以下条件满足时，才启用残差：

- 当前层不是第 0 层
- 上一层块分数存在
- 上一层和当前层的块数一致
- 当前 forward 的签名一致

否则退化为原始分数，不做残差。

## 5. 需要实现的代码点

### 5.1 `BlockWisePress`

新增：

- `cross_layer_score_residual_weight`
- 当前 forward 的签名缓存
- 每层的最终块分数字典

逻辑：

1. 进入 `analyze_blocks()` 时，根据 `cache_position + hidden_states.shape + key_len` 生成当前 forward 签名
2. 若签名变化，则清空跨层缓存，避免不同请求/不同 decode step 之间串扰
3. 计算当前层原始 `block_scores`
4. 若上一层分数可用，则做线性残差融合
5. 将融合后的分数写回当前层缓存，供下一层继续使用

### 5.2 `evaluate.py`

为了后续实验可复现，建议同步暴露：

- `cross_layer_score_residual_weight`

并把它写入结果目录名。

## 6. 风险与局限

### 风险 1

若某一层本身排序错误，残差可能把错误继续往后传播。

缓解：

- 先用较小权重 `0.2`

### 风险 2

如果某些层的真正关注点变化较大，过强的层间平滑可能压制当前层新信息。

缓解：

- 先保持“当前层主导，上层弱残差”

### 风险 3

如果块数不一致，强行残差会带来错误对齐。

缓解：

- 只有在块数一致时才做残差

## 7. 最小验证计划

### 第一阶段

只做最小但最有信息量的验证：

- 数据集：`RULER`
- 压缩率：`0.7`
- 采样比例：`0.5`
- 设备：`cuda:0`

比较对象：

- 旧版 `BlockWisePress`
- 新版 `BlockWisePress + cross-layer residual`
- `ChunkKV`

重点看：

- `niah_multikey_2`
- `niah_multikey_3`
- `niah_single_3`
- `qa_1`
- `qa_2`

### 第二阶段

如果第一阶段有效，再扩展到：

- `RULER 0.3`
- `LongBench`
- `Needle in a Haystack`

## 8. 成功标准

这次不要求全面追平 `ChunkKV`，只要满足下面任一条，就值得继续：

1. `RULER 0.7` 宏平均稳定回升
2. `niah_multikey_2/3` 明显回升
3. `niah_single_3` 和 `qa_1` 至少有一项显著改善，且没有整体明显副作用

## 9. 当前结论

相比继续在当前这版 `token correction` 上小修小补，跨层分数残差更值得优先尝试，因为：

- 它更贴近当前暴露出的“层间排序不稳定”问题
- 它更符合块级系统设计目标
- 实现和存储开销都更低
